## Project Overview

Pod Monitor is a Kubernetes resource monitoring dashboard. It displays pod CPU/memory requests, limits, and actual usage per namespace, with a Grafana-style dashboard including gauges, timeseries charts, and alert thresholds based on resource usage percentage.

## Commands

### Backend (Go)

```bash
cd backend
go build ./...          # Build
go run main.go          # Run locally (requires kubeconfig or in-cluster)
docker build -t pod-monitor-backend:latest .  # Build Docker image
```

### Frontend (React + Vite)

```bash
cd frontend
npm install             # Install dependencies
npm run dev             # Dev server (proxies /api/* to localhost:8080)
npm run build           # Production build
npm run lint            # ESLint
npm run preview         # Preview production build
```

### Kubernetes Deployment

```bash
# Apply all manifests in order
kubectl apply -f k8s/namespace.yaml

kubectl apply -f k8s/backend.yaml
```

## Architecture

### Data Flow

1. Frontend fetches `/api/namespaces` on mount
2. User selects a namespace and submits
3. Frontend requests `/api/resources?namespace={selected}`
4. Backend queries the Kubernetes API (pods) and Metrics API (actual usage)
5. Backend merges pod specs with metrics and returns JSON
6. Frontend parses K8s resource units, compares against thresholds, renders table

### Backend (`backend/main.go`)

Single-file Go HTTP server. Runs on port 8080. Uses in-cluster Kubernetes client (`k8s.io/client-go`) initialized via `initClients()`.

- `GET /api/namespaces` ‚Äî lists all namespaces
- `GET /api/resources?namespace=X` ‚Äî returns pods in namespace X with per-container resource requests, limits, and current usage; also saves a snapshot to SQLite (if enabled)
- `GET /api/history?namespace=X` ‚Äî returns up to 2000 historical pod snapshots from the last 7 days; `namespace` is optional
- `GET /api/dashboard/summary` ‚Äî returns aggregated cluster summary: pod counts by status, node counts, and active alerts (critical/warning); also triggers a SQLite snapshot for timeseries advancement
- `DELETE /api/admin/cluster/delete` ‚Äî removes a cluster from monitoring; requires admin JWT + password confirmation (bcrypt-verified)
- `/healthz` ‚Äî liveness probe endpoint

**Startup without clusters:** if no `KUBECONFIG` is set and not running in-cluster, the backend logs a warning and starts normally. Clusters are added at runtime via the admin UI (no `log.Fatal`).

**Alert thresholds** are evaluated in the backend based on usage % vs limit:
- `< 85%` ‚Äî OK
- `‚â• 85%` ‚Äî Warning
- `‚â• 90%` ‚Äî Critical

`DashSummary.Alerts` is a struct `{ Critical int; Warning int }`. Each `AlertPod` entry includes a `Severity` field (`"critical"` or `"warning"`).

**SQLite history** is optional and enabled by setting the `DB_PATH` environment variable. When enabled, each `/api/resources` call (and each `/api/dashboard/summary` call) saves a snapshot with a random `session_id`. A background goroutine purges records older than 7 days every 6 hours. SQLite is opened with `MaxOpenConns=1` (no concurrent writes).

If the metrics-server is unavailable, the backend gracefully returns pod spec data without usage metrics.

### Frontend (`frontend/src/App.jsx`)

Single React component with all state and logic. Uses `react-grid-layout` for the dashboard grid, which adapts to window size via `ResizeObserver`.

**Internationalisation (`frontend/src/i18n.js`)**
- All UI strings are defined in `frontend/src/i18n.js` with PT (default) and EN translations
- `useT(lang)` returns a `t(key)` function; falls back to PT then to the raw key
- `lang` state lives in the root `App` component, persisted via `localStorage('pm_lang')`
- A PT/EN toggle button in the header switches the language; preference survives page reloads
- `lang` is propagated as a prop to every sub-component that renders text: `LoginPage`, `AnalysisTab`, `DashWidget`, `TimeSeriesWidget`, `DashboardPage`, `HelpModal`, `UserManagementPage`, `DevAccessPicker`, `DevAccessModal`
- Help documentation (`HelpModal`) also switches language ‚Äî topics are generated by `getHelpTopics(lang)` which reads from `translations[lang].helpTopics`; topic-to-role mapping is kept separately in `HELP_TOPIC_ROLES`

**Module independence (Analysis, Logs and Top 10 tabs)**
- `AnalysisTab` has its own `anaCluster` / `anaNamespace` / `anaNsList` state and cluster+namespace selectors ‚Äî it no longer inherits the Monitor tab's selection
- The Logs tab has its own `logCluster` / `logNsList` state and a cluster selector in the UI ‚Äî namespace options (`logNsList`) are fetched from `/api/namespaces?cluster=logCluster` independently of the Monitor; `logCluster` is initialised from the global cluster on first open but can be changed freely
- The Top 10 tab has its own `top10Cluster` / `top10Pods` / `top10Loading` state ‚Äî data is fetched automatically via `/api/resources?cluster=X` when the tab is opened or the cluster selector changes; a Refresh button allows manual reload; `topOffenders` is derived from `top10Pods` (not from the Monitor's `pods`)
- `flatContainers` and `nsAggregated` continue to be derived from the Monitor's `pods` state (used by the Namespaces tab)
- All independent tabs call `/api/clusters` and `/api/namespaces` directly, so backend access-filtering (dev-role restriction) is respected automatically

**Tab naming**

**Dashboard (DashboardPage)**
- Auto-refreshes every 5 minutes via `setInterval(loadSummary, 300_000)`
- `refreshTick` counter increments on every summary load and is propagated to all widgets so timeseries charts update together
- Grid `rowHeight` is computed dynamically: `Math.max(60, Math.floor(availH / 7))`

**Gauge widgets (`SpeedGauge`)**
- Grafana speedometer style: SVG arcs (color zones) + absolutely positioned HTML overlay for the value and label
- The SVG/HTML split is necessary because SVG `<text>` cannot render JSX (e.g. `StatHover`)
- Color zones: green `< 75%`, yellow `75‚Äì90%`, red `‚â• 90%`
- Constants: `SG_VBW=200, SG_VBH=170, SG_CX=100, SG_CY=95, SG_R=68, SG_SW=14, SG_START=200¬∞, SG_SWEEP=220¬∞`

**Timeseries charts (TimeSeriesWidget)**
- Shows CPU and memory usage over time using Recharts `AreaChart`
- Time window zoom (`zoomMs`): 1h, 6h, 12h, 24h, 48h
- Tick granularity (`tickZoom`): 5m, 10m, 30m (double-click resets to auto)
- `calcTickValues(pts, intervalMin)` builds the explicit ticks array for the X-axis
- `fmtTick(iso, zoomMs)` formats ticks as `HH:MM` or `DD/MM HH:MM` depending on window size

**Alert thresholds** are evaluated in the backend (usage % vs limit). Frontend displays severity colors accordingly.

**Admin tab**
- Lists all monitored clusters with active badge
- Cluster deletion requires password confirmation modal (bcrypt-verified on backend)
- States: `delClusterModal`, `delClusterPwd`, `delClusterErr`, `delClusterLoading`
- Calls `DELETE /api/admin/cluster/delete` with `{ name, password }`

Resource unit parsing (millicores, Gi/Mi/Ki) is done on the frontend. The Vite dev server proxies `/api/*` to `localhost:8080` (see `vite.config.js`).

### Kubernetes (`k8s/`)

All resources live in the `pod-monitor` namespace. The backend uses a ServiceAccount (`pod-monitor-sa`) with a ClusterRole that grants read-only access to pods, namespaces, and pod metrics (`metrics.k8s.io`). Ingress routes `/api/*` to the backend and `/` to the frontend, with hostname `pod-monitor.local`.

### Deployment

Both components are containerized via multi-stage Dockerfiles. The frontend is served by Nginx in production using `frontend/nginx.conf`, which handles SPA routing.

The Nginx upstream is configurable via the `BACKEND_HOST` environment variable (default: `backend-svc` for Kubernetes, `backend` for docker-compose). The `nginx.conf` uses `${BACKEND_HOST}` as a template variable, substituted at container startup via `envsubst`. The `docker-compose.hub.yml` sets `BACKEND_HOST=backend` automatically. The Helm chart exposes this as `frontend.backendHost` (default: `backend-svc`).
